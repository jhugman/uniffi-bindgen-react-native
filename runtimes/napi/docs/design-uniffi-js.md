# Design: uniffi-js ŌĆö Unified FFI Player Architecture

## Summary

Replace uniffi-runtime-napi (Node.js only) and UBRN's generated C++ layer (React Native only) with a unified architecture where:

- A **shared TypeScript scaffolding** layer (generated by the bindgen) targets a platform-agnostic JS API
- Three **platform-specific players** implement that API: N-API (Node.js), JSI (React Native), WASM (browser/edge)
- Internal byte transfers (RustBuffer, ForeignBytes) minimize copying wherever the platform allows

The players are small, self-contained native modules. The complexity lives in the shared generated TypeScript above them.

## Motivation

Today, UBRN generates both TypeScript *and* C++ per UniFFI library. The C++ layer handles marshalling between JSI and the Rust FFI. This has costs:

- **Build complexity**: Downstream developers must compile generated C++ as part of their React Native build
- **Two code generators**: The TypeScript and C++ generators must stay in sync
- **No code sharing with Node.js**: uniffi-runtime-napi solves the same problem for Node but shares nothing with UBRN

By moving the marshalling into a pre-built native player module, the bindgen only needs to generate TypeScript ŌĆö and the same TypeScript (modulo import paths) works across all three platforms.

## The Player API

All three players expose the same JavaScript-facing API. This is the contract between the generated TypeScript and the native layer.

### Core FFI dispatch

- `open(path)` ŌĆö load a native library (or WASM module), return a handle
- `handle.register(definitions)` ŌĆö accept function/callback/struct definitions, return an object with callable methods
- `FfiType` ŌĆö type descriptor constants (`UInt8`, `Int32`, `RustBuffer`, `Callback(name)`, etc.)

The definitions object format (`symbols`, `functions`, `callbacks`, `structs`) is identical across all players. The generated TypeScript does not know or care which player it's talking to.

### Utility functions

Beyond FFI dispatch, the player provides utility functions that UBRN's generated TypeScript and runtime library depend on. These replace the "synthetic functions" that UBRN currently generates as per-crate C++ code.

**Object lifecycle management:**
- `bless(pointer, destructor)` ŌĆö pair a native pointer with a cleanup callback, return a guard object (`UniffiGcObject`). When the guard is collected or explicitly destroyed, the destructor is called. This is generic ŌĆö one implementation handles all object types from all crates. The generated TypeScript passes in the crate/type-specific `freePointer` function as the destructor.

**String conversion:**
- `stringToBytes(str)` ŌĆö convert a JS string to UTF-8 bytes (`Uint8Array`)
- `bytesToString(bytes)` ŌĆö convert UTF-8 bytes (`Uint8Array`) to a JS string

### Platform-specific utility implementations

These utilities exist on all players but are implemented differently:

| Utility | N-API / WASM | JSI |
|---------|-------------|-----|
| `bless` | TypeScript: `FinalizationRegistry` + guard object | Native C++: destructor guard via JSI host object lifecycle |
| `stringToBytes` | TypeScript: `TextEncoder` | Native C++: UTF-8 encoding via `jsi::String` API |
| `bytesToString` | TypeScript: `TextDecoder` | Native C++: UTF-8 decoding via `jsi::String` API |

Hermes (the React Native JS engine) does not support `FinalizationRegistry` or `TextDecoder` as of March 2026. The JSI player provides these natively in C++. The N-API and WASM players use standard JS APIs, so their implementations are thin TypeScript wrappers ŌĆö or the shared TypeScript runtime handles it directly.

### Platform-specific `open()` semantics

| Player | `open(path)` loads via | Symbol lookup |
|--------|----------------------|---------------|
| N-API  | `dlopen` | `dlsym` |
| JSI    | `dlopen` | `dlsym` |
| WASM   | `WebAssembly.instantiate` | `instance.exports[name]` |

On iOS, where `dlopen` of third-party dynamic libraries is blocked in App Store builds, the Rust library is statically linked. `dlsym(RTLD_DEFAULT, name)` resolves symbols from the main binary. The `path` argument may be ignored or used as a sentinel.

## Player Implementations

### N-API Player (Rust)

This is the current uniffi-runtime-napi, largely unchanged. It is a napi-rs native addon that uses libffi for dynamic dispatch.

- **Language:** Rust
- **Distribution:** npm (prebuilt binaries via napi-rs toolchain)
- **Runtime:** Node.js

### JSI Player (C++ frontend, Rust core)

A React Native Turbo Module using the JSI C++ API. The C++ layer is thin ŌĆö it reads JS values, calls into a shared Rust core via `extern "C"`, and wraps results as JSI values.

- **Language:** C++ (JSI glue) + Rust (libffi dispatch core)
- **Distribution:** npm + CocoaPods + Gradle (standard React Native native module)
- **Runtime:** Hermes (React Native)

The Rust core is a static library exposing C functions. The C++ layer links against it. This means:

- CIF construction, `cif.call` dispatch, callback trampoline setup live in Rust
- JS value reading/writing, `jsi::HostFunction` creation, `CallInvoker` thread dispatch live in C++

### WASM Player (Pure TypeScript)

No native code. WASM has a fixed ABI ŌĆö JS calls WASM exports directly. No libffi, no dlopen.

- **Language:** TypeScript
- **Distribution:** npm
- **Runtime:** Any JS engine with WebAssembly support (browser, Node.js, Deno, edge workers)

The WASM player:
1. Loads the `.wasm` file via `WebAssembly.instantiate`
2. Wraps exports in the same API shape that `register()` returns
3. Handles RustBuffer marshalling by reading/writing WASM linear memory directly

No wasm-bindgen is involved. UniFFI's exported functions already use primitive C types (i32, i64, f32, f64) which map directly to WASM value types. The generated TypeScript handles higher-level marshalling (strings, enums, objects to serialized bytes) the same way it does for the native players.

**WASM memory caveat:** `Uint8Array` views over WASM linear memory are invalidated when memory grows. The player must re-create views after any call that might trigger allocation (i.e., any call into WASM). Short-lived views created per-call are safe.

## Minimizing Byte Copies

### Current state (uniffi-runtime-napi)

Every RustBuffer crossing copies twice:

1. **JS to Rust:** `Uint8Array` is copied into Rust-allocated memory via `rustbuffer_from_bytes`
2. **Rust to JS:** `RustBufferC` data is copied into a new JS `Uint8Array`, then `rustbuffer_free` releases the Rust allocation

### Goal: zero-copy where the platform allows

#### WASM Player ŌĆö zero-copy naturally

WASM linear memory is a JS `ArrayBuffer`. Both JS and WASM operate on the same memory:

- **JS to WASM:** Write bytes into WASM linear memory via a `Uint8Array` view, pass the pointer (offset) as an i32. No copy.
- **WASM to JS:** Create a `Uint8Array` view over the returned pointer/length. No copy.

The only constraint is view lifetime ŌĆö views are invalidated by memory growth. For RustBuffer returns, the pattern is:
1. Call the WASM function, get back pointer + length
2. Create a view, copy into a *new* `Uint8Array` (one copy ŌĆö unavoidable if the buffer will be freed)
3. Call `rustbuffer_free`

Or, for read-only access that doesn't outlive the call: create a view and consume it immediately (true zero-copy).

#### N-API Player ŌĆö zero-copy via external ArrayBuffer

N-API provides `napi_create_external_arraybuffer`, which wraps native-owned memory as a JS `ArrayBuffer` with a finalize callback. This enables:

- **Rust to JS (RustBuffer return):** Wrap the `RustBufferC`'s data pointer as an external `ArrayBuffer`. The finalize callback calls `rustbuffer_free`. Zero copies ŌĆö JS reads Rust-allocated memory directly, and it's freed when the JS `ArrayBuffer` is garbage collected.

- **JS to Rust (RustBuffer argument):** If the `Uint8Array` is already backed by Rust-allocated memory (e.g., a buffer returned from a previous call), pass its pointer directly via `ForeignBytes`. One fewer copy. If it's a fresh JS buffer, one copy via `rustbuffer_from_bytes` is still needed.

**Tradeoff:** External ArrayBuffers tie Rust memory lifetime to JS garbage collection. This is fine for buffers that are consumed quickly but could delay freeing for long-lived buffers. An explicit `dispose()` or `using` pattern could mitigate this if needed.

#### JSI Player ŌĆö zero-copy via MutableBuffer

JSI's `jsi::ArrayBuffer` accepts a custom `MutableBuffer` that wraps native memory. Same pattern as N-API: wrap `RustBufferC` data, free on destruction. The C++ JSI layer manages this naturally.

#### Internal buffers (between player layers)

Within the N-API and JSI players, the Rust core and the JS-engine frontend communicate via pointers. No intermediate copies are needed ŌĆö the frontend passes raw byte pointers into the Rust core, and the core returns pointers back. The copy-minimization is about the JS-to-native boundary, not internal to the player.

## SharedByteBuffer (future extension)

A new type for user-facing zero-copy buffers, where the application writes bytes, passes the buffer to Rust for processing, and reads the result from the same buffer. This is independent of the core player architecture and can be added later.

### Without uniffi-rs support

The Rust library exports functions that take raw pointers and lengths. The bindgen (UBRN) annotates certain parameters as `SharedByteBuffer` via configuration or custom attributes. The generated TypeScript allocates shared memory through the player and passes pointers.

- Rust side: `fn process(data: *mut u8, len: i32)` ŌĆö raw pointers, more `unsafe`
- Player allocates memory appropriate to the platform (WASM linear memory / external ArrayBuffer / JSI MutableBuffer)
- No changes to uniffi-rs required

### With uniffi-rs support

UniFFI gains a first-class `ByteBuffer` type that lowers to `(*mut u8, len)` in the FFI layer. Library authors get safe Rust APIs (`&mut ByteBuffer` with bounds-checked slice access). Bindgens get type metadata automatically.

### Recommendation

Start without uniffi-rs support. Prove the ownership model across all three platforms. Use that experience to inform an upstream proposal.

## Code Sharing Between N-API and JSI Players

The N-API and JSI players both use libffi for dynamic dispatch. The shared Rust code includes:

- `FfiTypeDesc` enum and parsing
- CIF construction (`ffi_type_for`, building CIF from type descriptors)
- `cif.call` dispatch (`call_with_ret_type`)
- Type-erased scalar marshalling (`boxed_to_arg`)
- Callback trampoline core (CIF for callbacks, trampoline function signature)
- `RustBufferC` and `RustCallStatusC` struct definitions

This becomes a **core Rust crate** (`uniffi-ffi-core` or similar) exposing `extern "C"` functions. The N-API player calls it from Rust directly (crate dependency). The JSI player's C++ frontend calls it via C FFI (linked static library).

What remains platform-specific:

**N-API frontend (Rust):**
- Read JS values via `napi::JsNumber`, `JsBigInt`, etc.
- Create JS values via `env.create_uint32()`, `env.create_bigint_from_u64()`, etc.
- Create callable JS functions via `env.create_function_from_closure()`
- Thread-safe callback dispatch via `ThreadsafeFunction`
- Stop GC from collecting pointers via `napi_create_reference`

**JSI frontend (C++):**
- Read JS values via `jsi::Value::asNumber()`, `asBigInt()`, etc.
- Create JS values via `jsi::Value(runtime, n)`, `jsi::BigInt::fromUint64()`, etc.
- Create callable JS functions via `jsi::Function::createFromHostFunction()`
- Thread-safe callback dispatch via `CallInvoker`
- Keep JS pointers from being GC'd via jsi::Object global references

The WASM player shares no Rust code with the other two ŌĆö it is pure TypeScript. But it is also the simplest, since it has no libffi, no threading, and no native memory management beyond WASM linear memory views.

## Testing Strategy

All three players implement the same JS API, so the same test suite should run against all three.

### Test runtimes

| Player | Test runtime | How |
|--------|-------------|-----|
| N-API  | Node.js | `node --test` (or vitest/jest) ŌĆö current approach |
| JSI    | Hermes | Run tests via a Hermes CLI binary, or via a minimal React Native test harness |
| WASM   | Node.js + WASM | Same Node.js runner, but load WASM player instead of N-API player |

### Test structure

```
tests/
  shared/              # Platform-agnostic test cases
    scalars.test.mjs
    rust_buffer.test.mjs
    callbacks.test.mjs
    threading.test.mjs  # N-API + JSI only (WASM is single-threaded)
    ...
  fixtures/
    libmylib.dylib     # Prebuilt test Rust library (native)
    mylib.wasm          # Prebuilt test Rust library (WASM)
  harness/
    napi.mjs            # Loads N-API player, exports test helpers
    jsi.mjs             # Loads JSI player, exports test helpers
    wasm.mjs            # Loads WASM player, exports test helpers
```

Each test file imports from a harness module that provides the player. The harness is selected at test invocation time (e.g., via an env var or test runner config). The shared tests don't import the player directly.

### What to test per-player

- **Shared tests (all players):** Scalar calls, RustBuffer round-trips, callbacks, VTable structs, error propagation, megazord (multi-crate) registration.
- **N-API/JSI only:** Cross-thread callback dispatch (blocking and non-blocking), thread safety.
- **WASM only:** Memory growth behavior (views invalidated after growth), WASM-specific import/export patterns.
- **Per-player unit tests:** Internal implementation details specific to each player (e.g., external ArrayBuffer lifecycle for N-API, CallInvoker dispatch for JSI).

### Test Rust library

A single Rust library (`uniffi-test-fixtures` or similar) compiled to both native (`.dylib`/`.so`) and WASM (`.wasm`). It exports functions exercising all supported patterns: scalars, buffers, callbacks, async, errors.

## Repository Structure

This design is intended to work whether the code lives in a standalone repository or is merged into UBRN. The decision can be deferred.

### Option A: Standalone monorepo (current)

```
uniffi-js/                   # renamed from uniffi-runtime-napi
  crates/
    uniffi-ffi-core/         # Shared Rust: libffi dispatch, CIF, type-erased marshalling
    uniffi-runtime-napi/             # N-API frontend (Rust, napi-rs)
    uniffi-jsi-core/         # JSI backend Rust core (extern "C" API)
  cpp/
    uniffi-jsi/              # JSI frontend (C++, calls into uniffi-jsi-core)
  ts/
    uniffi-wasm/             # WASM player (pure TypeScript)
  tests/                     # Shared test suite
  docs/
```

- Published as separate npm packages: `uniffi-runtime-napi`, `uniffi-jsi`, `uniffi-wasm`
- UBRN depends on these as npm packages
- Independent release cycle

### Option B: Merged into UBRN

```
uniffi-bindgen-react-native/
  crates/
    uniffi-bindgen/          # Existing: the Rust codegen CLI
    uniffi-ffi-core/         # Shared Rust core (from this project)
    uniffi-runtime-napi/             # N-API frontend
    uniffi-jsi-core/         # JSI backend Rust core
  cpp/
    uniffi-jsi/              # JSI C++ frontend
  ts/
    uniffi-wasm/             # WASM player
  tests/
    player/                  # Player test suite (from this project)
```

- Tighter integration with the codegen
- Single release cycle
- Larger repo, more build complexity

### Recommendation

Start standalone (Option A). Merge later if the integration overhead of separate repos outweighs the benefits. The key constraint is that the player API (`open`/`register`/`FfiType`) must be stable enough for UBRN to depend on ŌĆö and that API is already well-defined.

## Packaging and Distribution

### The player package: `uniffi-bindgen-react-native`

The player and the codegen CLI ship together as a single npm package. This is UBRN today, extended with the player architecture.

```
uniffi-bindgen-react-native/
  bin/                            # Codegen CLI
  players/
    jsi/                          # JSI player (C++ source + prebuilt Rust .a/.so)
    napi/                         # N-API player (prebuilt native addon)
    wasm/                         # WASM player (pure TypeScript)
  android/
    build.gradle                  # Autolinking: compiles C++ JSI glue, links Rust core
  ios/
    uniffi-jsi.podspec            # Autolinking: compiles C++ JSI glue, links Rust core
  package.json
```

The `package.json` uses conditional exports so generated TypeScript can import from a single path:

```json
{
  "exports": {
    "./player": {
      "react-native": "./players/jsi/index.js",
      "node": "./players/napi/index.js",
      "browser": "./players/wasm/index.js",
      "default": "./players/napi/index.js"
    }
  }
}
```

Generated TypeScript imports generically:

```typescript
import { open, FfiType } from 'uniffi-bindgen-react-native/player';
```

Metro (React Native), Node.js, and browser bundlers (webpack, vite) each resolve the correct player automatically via the conditional export.

### React Native autolinking

React Native's autolinking system scans `node_modules/` for packages with a `podspec` or `android/build.gradle` and automatically integrates them into the app build. No manual linking required.

When a consumer installs a library that depends on `uniffi-bindgen-react-native`:

1. `npm install my-cool-library` installs both the library and UBRN (transitive dep)
2. `pod install` (iOS) / Gradle sync (Android) finds UBRN's native module config
3. Autolinking compiles the generic C++ JSI glue and links the prebuilt Rust player core
4. The JSI TurboModule is registered automatically at app startup
5. At runtime, the generated TypeScript calls `open()` and the player loads the client library

The consumer never sees Rust, C++, Kotlin, or Swift. It looks like any other npm package.

### The JSI player's native registration

The Kotlin (Android) and ObjC (iOS) registration code is entirely generic ŌĆö it registers a single TurboModule called `UniffiJsi` that receives `jsi::Runtime` and `CallInvoker` from React Native. This code is part of the UBRN package and is compiled by the consumer's app build via autolinking. No per-library native registration is needed.

### Client library repo layout

The library author's repo is a Cargo workspace with a thin `js/` directory for UBRN config. No generated code or prebuilt binaries are checked in ŌĆö they are build artifacts, created at publish time.

```
my-project/
  Cargo.toml                    # [workspace]
  crates/
    my-core/                    # Business logic (pure Rust)
    my-ffi/                     # UniFFI-exposed API (depends on my-core)
  js/
    package.json                # devDependency on UBRN, scripts for generate/publish
    uniffi.toml                 # UBRN config: which crate, what targets, package names
  .gitignore                    # ignores js/generated/, js/prebuilt/, js/packages/
```

The `js/package.json`:

```json
{
  "name": "my-cool-library",
  "private": true,
  "devDependencies": {
    "uniffi-bindgen-react-native": "^2.0.0"
  },
  "scripts": {
    "generate": "uniffi-bindgen-react-native generate --ts-only",
    "build": "uniffi-bindgen-react-native build --all-targets",
    "publish": "uniffi-bindgen-react-native publish --all"
  }
}
```

During local development, the author runs `npm run generate` to create TypeScript in `js/generated/` for testing. This directory is gitignored.

### Two consumption modes

**Published to npm (external consumers):**

Consumers install prebuilt packages from npm. No Rust toolchain needed. They have no idea the library is Rust.

```bash
npm install @my-cool-library/react-native-android
```

**Git dependency (internal consumers / source builds):**

Consumers with the Rust toolchain use a git dependency. UBRN generates TypeScript and cross-compiles Rust as part of the consumer's app build.

```json
{
  "dependencies": {
    "my-cool-library": "git://github.com/org/my-project.git#main"
  }
}
```

No prebuilt binaries, no generated code in the repo. Everything is built on demand.

### Publish pipeline

Publishing is fully automated by UBRN's CLI. The library author (or CI) runs `npm run publish`, which:

1. Cross-compiles the Rust crate for all configured targets (iOS arm64, Android arm64/x86_64, macOS arm64/x86_64, Linux x86_64, wasm32)
2. Generates TypeScript from the UniFFI interface
3. Assembles platform-specific npm packages in a temporary `js/packages/` directory ŌĆö each containing the generated TypeScript and the relevant binaries
4. Publishes each package to npm

The assembled packages are ephemeral ŌĆö created in CI, published, never committed. The repo stays clean.

A typical CI pipeline (GitHub Actions):

```yaml
jobs:
  publish:
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: aarch64-apple-ios, aarch64-linux-android, ...
      - run: cd js && npm ci
      - run: cd js && npm run publish
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

### Client library distribution tiers

Libraries can be distributed at three levels, depending on how much build effort the consumer is willing to accept.

#### Prebuilt (most consumers)

Each target ecosystem gets its own package with generated TypeScript and prebuilt binaries. The consumer installs one and it just works.

```
my-cool-library-react-native              # Meta-package: depends on -android + -ios
my-cool-library-react-native-android      # Generated TS + Android .so files (all ABIs)
my-cool-library-react-native-ios          # Generated TS + iOS .a / xcframework
my-cool-library-node                      # Generated TS + platform-specific optional deps
my-cool-library-wasm                      # Generated TS + .wasm file
```

Alternatively, scoped packages:

```
@my-cool-library/react-native
@my-cool-library/react-native-android
@my-cool-library/react-native-ios
@my-cool-library/node
@my-cool-library/wasm
```

Each prebuilt package contains:
- The same generated TypeScript (identical across platforms)
- Prebuilt binaries for that platform only
- A runtime dependency on `uniffi-bindgen-react-native` (for the player)

For Node.js, the platform-specific binary packages use the napi-rs optional dependency pattern ŌĆö npm only downloads the binary matching the developer's machine.

React Native iOS and Android are split because `npm install` happens on the dev machine, not the target device. A developer building only for Android doesn't need iOS binaries and vice versa.

UBRN's CLI automates publishing all packages from a single source:

```bash
uniffi-bindgen-react-native publish --all
```

#### Source (advanced / custom targets)

No npm publish needed. The consumer uses a git dependency and builds from source:

```json
{
  "dependencies": {
    "my-cool-library": "git://github.com/org/my-project.git#main"
  }
}
```

The consumer needs the Rust toolchain installed. UBRN generates TypeScript and cross-compiles Rust as part of the consumer's app build. No prebuilt binaries, no generated code in the repo.

This is essentially what UBRN does today. With the player architecture, it becomes the opt-in path for developers who need custom targets or want to build from source, rather than the only option.

### Client library package structure (prebuilt, React Native Android example)

```
my-cool-library-react-native-android/
  package.json
  generated/
    index.ts                    # Public API
    mylib.ts                    # Scaffolding (open + register + typed wrappers)
  android/
    build.gradle                # Copies .so into jniLibs
    src/main/jniLibs/
      arm64-v8a/libmylib.so
      x86_64/libmylib.so
```

The `build.gradle` is minimal ŌĆö it only ensures the prebuilt `.so` files end up in the APK. No compilation, no C++, no Kotlin. Autolinking picks it up.

The `package.json`:

```json
{
  "name": "my-cool-library-react-native-android",
  "main": "generated/index.ts",
  "dependencies": {
    "uniffi-bindgen-react-native": "^2.0.0"
  }
}
```

### Client library package structure (prebuilt, iOS example)

```
my-cool-library-react-native-ios/
  package.json
  generated/
    index.ts
    mylib.ts
  ios/
    my-cool-library.podspec     # Links .a with -force_load
```

The podspec only links the prebuilt static library. No source compilation.

### How `open()` resolves libraries per platform

| Platform | `open('mylib')` resolves to |
|----------|----------------------------|
| Android  | `dlopen("libmylib.so")` ŌĆö found in app's native lib dir via jniLibs |
| iOS      | `dlsym(RTLD_DEFAULT, ...)` ŌĆö symbols statically linked into main binary |
| Node.js  | `dlopen("path/to/libmylib.dylib")` ŌĆö path resolved from optional dep package |
| WASM     | `WebAssembly.instantiate(bytes)` ŌĆö `.wasm` loaded from package |

## Open Questions

1. **iOS static linking:** How does the JSI player resolve symbols when the Rust library is statically linked? Likely `dlsym(RTLD_DEFAULT, name)`, but needs validation on-device.

2. **Hermes testing:** Is there a standalone Hermes CLI suitable for running the shared test suite, or do we need a minimal React Native test app?

3. **WASM compilation target:** Should the test Rust library target `wasm32-unknown-unknown` or `wasm32-wasi`? Affects available stdlib and async support.

4. **Async (Rust futures):** The current design covers synchronous FFI calls. Async support (via `uniffiRustCallAsync` and continuation callbacks) is needed for both JSI and WASM players. The callback trampoline machinery supports this but the polling/completion integration is not yet designed.

5. **SharedByteBuffer ownership:** Who allocates, who frees? Does the player manage a pool, or is it one-shot? Does `using`/`Symbol.dispose` fit naturally?

6. **BigInt on Hermes:** Hermes has BigInt support but it was added relatively recently. Verify it works for all Handle/u64/i64 patterns.

7. **WASM 64-bit integers:** WASM natively supports i64, but the JS-WASM boundary historically converted i64 to BigInt. This is now standard but older engines may not support it. Define minimum engine requirements.
